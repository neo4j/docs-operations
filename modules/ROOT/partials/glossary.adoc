

[glossary]

[[allocator]]allocator::
A component in the cluster that allocates databases to servers according to the topology constraints specified and an allocation strategy.

[[asynchronous-replication]]asynchronous replication (cluster), see also <<term-synchronous-replication, synchronous replication>>::
Asynchronous replication is used by secondary copies to poll for new transactions, which means they cannot be guaranteed to have received the most recent transactions.
This enables efficient scale-out of read-performance.

[[aura-instance]]Aura instance::
A fully-managed DBMS represented by a single instance ID, that is running in the Neo4j Aura cloud.

[[auto-commit-transaction]]auto-commit transaction::
An automatically committed transaction that contains a single query.

[[bolt-protocol]]Bolt protocol::
Bolt is a protocol used for interaction between Neo4j instances and drivers.

[[bookmark]]bookmark, see also: <<term-causal-consistency, causal consistency>>::
A marker the client can request from the cluster to ensure that it is able to read its own writes so that the application's state is consistent and only databases that have a copy of the bookmark are permitted to respond.

[[category]]category (Bloom)::
A category is based on a node label and is defined in a Perspective as a way of visually distinguishing nodes with the same label(s).

[[causal-consistency]]causal consistency, see also <<term-bookmark, bookmark>>::
All servers in a cluster agree on the order in which transactions take place.
The position of a server on the causal chain can be guaranteed using a bookmark.

[[cluster]]cluster, see also: <<term-fault-tolerance, fault tolerance>>, <<term-standalone, standalone>>::
A Neo4j DBMS that spans multiple servers working together to increase fault tolerance and/or read scalability.
Databases on a cluster may be configured to replicate across servers in the cluster thus achieving read scalability or high availability.

[[client-application]]client application::
Software that interacts with a Neo4j server.

[[commit]]commit::
A commit is the successful completion of a transaction, which ensures durability of any changes made.
For more details, visit link:https://neo4j.com/docs/operations-manual/current/database-internals/transaction-management/[Operations Manual -> Transaction management].

[[composite-database]]Composite database, see also: <<term-fabric, fabric>>::
Composite databases are the means to access partitioned graph data with a single Cypher query.

[[constraint]]constraint::
Constraints are sets of data modeling rules that ensure the data is consistent and reliable.

[[core-server]]Core server (Neo4j 4), see also: <<term-primary-vs-secondary, primary vs secondary>>::
A server in a cluster operating in read/write mode.
This is replaced in Neo4j 5 by the database-level configuration of primary and secondary databases.

[[cypher]]Cypher®::
Neo4j's graph query language.

[[data-model]]data model::
A data model defines how information is organized in a database.
A good data model will make querying and understanding your data easier.
In Neo4j, the data models have a graph structure.

[[database]]database, see also: <<term-database-vs-graph, database vs graph>>::
A database is a container used by the DBMS to manage and store graph data.
The physical structure of data is controlled by the database.

[[database-vs-graph]]database vs graph::
Databases are the physical containers of graph data.
Graphs are the logical structure of data in Neo4j.

[[database-management-system]]Database Management System::
Database Management System, or DBMS, capable of managing multiple databases.
A DBMS may run on a single server, or span several servers configured as a cluster.

[[database-schema]]database schema, see also: <<term-data-model, data model>>::
The prescribed property existence and datatypes for nodes and relationships.

[[deallocate]]deallocate (cluster)::
An act of removing a database from a server or a server from a cluster without loss of data or reduced fault tolerance.

[[degree]]degree (of a node)::
The number of relationships of a specific node; loops are counted twice.

[[disaster-recovery]]disaster recovery (cluster)::
A manual intervention to restore availability of a cluster, or databases within a cluster.

[[driver]]driver::
A software library that provides access to Neo4j from a particular programming language.

[[election]]election (cluster), see also <<term-raft-protocol, Raft protocol>>::
In the event that the Raft leader becomes unresponsive, followers automatically trigger an election and vote for a new leader.

[[entity]]entity::
A node or a relationship.

[[expression]]expression (Cypher)::
A component of a Cypher query which produces values.
It may be used in projections, as a predicate, or when setting properties on graph elements.

[[fabric]]fabric, see also: <<term-composite-database, composite database>>::
Fabric is the architectural design of a unified system that provides a single access point to local or distributed graph data.

[[fault-tolerance]]fault tolerance (cluster), see also: <<term-primary, primary>>::
A guarantee that a cluster can maintain a database's persistence and availability in the event of one or more servers failing.

[[follower]]follower (cluster), see also <<term-leader, leader>>::
A primary copy of a database acting as a follower, receives and acknowledges synchronous writes from the leader.

[[generative-ai]]Generative AI (GenAI)::
A type of artificial intelligence (AI) system that generates text, images, or other media in response to prompts.

[[gpt]]Generative Pre-Trained Transformer (GPT)::
A type of GenAI model that combines two forms of training to produce foundation models.
Specifically:
* Pre-Training: Training general purpose capabilities on vast quantities of data.
* Fine-Tuning: Training a finite number of supervised ML tasks on a small amount of hand-picked data.

[[graph]]graph, see also: <<term-database-vs-graph, database vs graph>>::
A logical representation of a set of nodes where some pairs are connected by relationships.

[[index]]index::
Data structure that improves read performance of a database.

[[knowledge-graph]]knowledge graph::
A specific type of graph that has an organizing principle so that a user (or a computer system) can reason about the underlying data.
The organizing principle provides an additional layer of structure that adds context to support knowledge discovery.

[[label]]label::
Marks a node as a member of a named and indexed subset.
A node may be assigned zero or more labels.

[[language-model]]Language Model (LM)::
An ML approach that models the probability distribution over a sequence of words.
Predicts the probabilities of next word/character in a sequence.
Applications in GenAI as well as embedding, classification, and other ML tasks.

[[large-language-model]]Large Language Model (LLM)::
LMs consisting of large neural networks (billions of parameters) trained on large quantities of data often using self-supervised/semi-supervised approaches.
Trained for general tasks and currently seen as the "GenAI for language/text".

[[large-language-model]]LLM Hallucination::
Language models generate text that is incorrect, nonsensical, or unreal.
* Appear to answer questions confidently even if they do not have facts.
* May provide contradicting or inconsistent responses to similar prompts.

[[leader]]leader (cluster), see also: <<term-follower, follower>>::
A single primary copy of a database is designated as the leader.
It receives all write transactions from clients and replicates writes synchronously to followers and asynchronously to secondary copies of the database.

[[main-database]]main database, see also: <<term-tool-asset-database, tool asset database>>::
In terms of Neo4j Enterprise Studio, the database(s) containing the user's data.
Can exist in the same Neo4j deployment as the tool asset database.

[[motif]]motif, see also: <<term-path, path>> and <<term-pattern, pattern>>::
A description of a specific pattern within a graph.

[[node]]node, see also: <<term-relationship, relationship>>::
A node represents an entity or discrete object in your graph data model.
Nodes can be connected by relationships, hold data in properties, and are classified by labels.

[[operator]]operator::
A symbol representing a mathematical or logical operation.

[[parameter]]parameter::
Named value provided when running a Cypher statement.

[[path]]path::
A sequence of nodes and the relationships connecting them, that does not contain duplicate relationships.
Several paths can match a pattern.

[[pattern]]pattern, see also: <<term-motif, motif>>::
A specific arrangement of nodes and relationships that can be matched in a graph.
A pattern follows a motif.

[[perspective]]perspective (Bloom)::
A Perspective defines a certain business view or domain that can be found in the target Neo4j graph.
A single Neo4j graph can be viewed through different Perspectives, each tailored for a different business purpose.

[[primary]]primary (cluster), see also: <<term-secondary, secondary>>::
A copy of the database that is able to process write transactions and is eligible to be elected as a leader.
It participates in fault tolerant writes as it is part of the majority required to acknowledge and commit write transactions.

[[primary-vs-secondary]]primary vs secondary (cluster)::
In a cluster, databases can operate in either primary or secondary mode.
Primary databases are able to process write and read transactions, ensuring fault tolerance.
Secondary databases are replicated asynchronously from primaries, and their main purpose is to provide read scaling within the cluster.

[[project]]project (Aura), see also <<term-tenant, tenant>>::
An isolated environment in the unified Aura console that contains its own database instances, configurations, and resources.
Preceded by *tenant* in the classic Aura console.

[[property]]property::
Properties are key-value pairs that are used for storing data on nodes and relationships.

[[query]]query (Cypher)::
A statement that retrieves or writes information to a database.

[[raft-group]]Raft group::
A group of servers that are participating in hosting a particular database in primary mode.

[[raft-group-member]]Raft group member::
A server that is participating in a Raft group.
A server can be a member of one or more groups.

[[raft-log]]Raft log::
A shared log between all Raft group members that is guaranteed to be consistently updated and viewed by those members.
The log contains both database data and operational state of the Raft group.

[[raft-protocol]]Raft protocol::
The networking mechanism that enables a database to replicate its data across multiple servers to give high availability for accessing the data and high durability to the data stored.

[[read-replica]]Read Replica (Neo4j v4), see also: <<term-core-server, Core Server>>::
A server in a cluster operating in read-only mode.
This is replaced in Neo4j 5 by the database-level configuration of primary and secondary databases.

[[read-scaling]]read scaling, see also: <<term-primary-vs-secondary, primary vs secondary>>::
Distributing query load by creating additional database copies hosted in secondary mode (read-only).

[[relationship]]relationship, see also: <<term-node, node>>::
A relationship represents a connection between nodes in your graph data model.
Relationships connect a source node to a target node, hold data in properties, and are classified by type.

[[secondary]]secondary (cluster), see also: <<term-primary, primary>>::
An asynchronously replicated copy of the database that provides read scaling within the cluster.

[[seed]]seed (cluster)::
A seed is a database dump or a full backup used  to create a database on a cluster.
This is sometimes called _seeding_.

[[server]]server::
A physical machine, a virtual machine, or a container running an instance of Neo4j.
Servers can be standalone or part of a cluster.

[[session]]session::
A causally linked sequence of transactions.

[[session-consistency]]session consistency, see also: <<term-causal-consistency, causal consistency>>::
An alternative name for Neo4j’s causal consistency.

[[standalone]]standalone, see also: <<term-server, server>>::
A single server running Neo4j and not part of a cluster.

[[synchronous-replication]]synchronous replication (cluster), see also <<term-asynchronous-replication, asynchronous replication>>::
Synchronous replication requires the leader primary to replicate a transaction and block the commit until a quorum of the follower primaries acknowledges that the transaction is successfully replicated.
Once the transaction is  replicated, the commit is allowed to proceed.
This ensures data durability and consistency within the cluster.

[[system-database]]system database::
A database used by Neo4j to store system information.

[[tenant]]tenant (Aura), see also <<term-project, project>>::
An isolated environment in the classic Aura console that contains its own database instances, configurations, and resources.
Replaced by *project* in the unified Aura console.

[[tool-asset-database]]tool asset database, see also <<term-main-database, main database>>::
In terms of Neo4j Enterprise Studio, the database where tools' assets are stored.
This can be in the same Neo4j deployment as the main database(s) or in a separate deployment.

[[topology]]topology (cluster)::
A configuration that describes how the copies of a database should be spread across the servers in a cluster, see primary mode and secondary mode.

[[transaction]]transaction::
A transaction comprises a unit of work performed against a database.
It is treated in a coherent and reliable way, independent of other transactions.
Transactions comply with the ACID consistency model (atomic, consistent, isolated, and durable).
